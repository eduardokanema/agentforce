## Part 3: Specialized Agent Analysis

### 3A. Architecture & Performance Analysis

**Mission State Machine: Tick-Based Decoupling**

The AgentForce core implements a **tick-based state machine** that decouples orchestration from execution:

- **MissionEngine** (engine.py): Supervisor that drives missions via `tick()` cycles, returning action lists (WorkerDelegation, ReviewerDelegation, HumanIntervention)
- **Task Lifecycle**: PENDING → IN_PROGRESS → COMPLETED → REVIEWING → REVIEW_APPROVED (with retry and escalation paths)
- Caller dispatch: Results fed back via `apply_worker_result()`, `apply_reviewer_result()`, `apply_human_resolution()`
- **Key Insight**: This decoupling enables distributed execution and local/remote dispatch flexibility

**Task Dependency Model**

- Enforced in `dispatchable_tasks()`: Only tasks with **all dependencies in REVIEW_APPROVED** status can dispatch
- Circular dependency detection at spec load time (DFS-based cycle detection)
- Prevents out-of-order execution; ensures downstream tasks have upstream context
- **Example**: In http-server mission, Task 02 (metrics) and Task 03 (error handling) can run parallel only after Task 01 (scaffolding) approves

**Retry & Approval Thresholds**

| Cap | Default | Enforcement in engine.py |
|-----|---------|--------------------------|
| `max_retries_global` | 3 | Total retries across all tasks |
| `max_retries_per_task` | 3 | Per-task retry limit |
| `max_wall_time_minutes` | 120 | Mission execution deadline |
| `max_human_interventions` | 2 | Human escalation budget |
| `max_concurrent_workers` | 3 | Parallel dispatch limit |

Caps checked on every `tick()`: if hit, mission → FAILED, all pending dispatch stops.

**Connector System & Session Caching**

| Connector | Session Caching | Mechanism |
|-----------|-----------------|-----------|
| OpenCode | ✅ Enabled | Supports `--session <id> --continue` for warm-start |
| Claude | ❌ Disabled | Internal state, no stateful session support |
| OpenRouter | ❌ Disabled | Stateless HTTP API |
| Codex | ❌ Disabled | CLI stateless |

**Session Caching Benefit**: Warm-start context in retries reduces re-explaining; estimated 15-25% token savings on Task 01 retry (evident in the 269s→128s pattern if caching were enabled).

**Review Loop Mechanics**

1. Worker generates output; engine marks task COMPLETED
2. ReviewerDelegation created with task spec + worker output + acceptance criteria
3. Reviewer scores against checklist (0-10 scale)
4. Result: `(approved: bool, score: 0-10, blocking_issues: [str])`
5. If approved: REVIEW_APPROVED, dispatch dependent tasks
6. If rejected + retries available: retry worker with feedback
7. If rejected + no retries: escalate to NEEDS_HUMAN

**Review Scoring Calibration** (observed in http-server mission):
- 4-6: Implementation incomplete or acceptance criteria gaps
- 7-8: Solid work with minor polish issues
- 9-10: Exceeds expectations, production-ready

The 9.0/10 uniform score across all 5 tasks suggests consistent reviewer expectations and delivery quality, not score inflation.

## Part 5: Implications for Phase 0 Backend Server

### Architectural Lessons Learned

#### 1. Early Shape Decisions Are Load-Bearing
Task 01 (scaffolding) required 3 attempts and consumed the most time (269s). This suggests architectural choices made early have outsized impact. **Recommendation**: Pre-define server skeleton, base interfaces, and middleware hooks before workers begin.

#### 2. Middleware Patterns Should Be First-Class
Token tracking, instrumentation, connector bridging — these are not features to bolt on later. They should be baked into the server core from the beginning. Integration tests should validate these paths, not just the happy path.

#### 3. Connector Boundaries Must Be Clean
Request/response contracts, error mapping, metrics collection hooks, rate-limiting — all must be designed upfront. Unclear boundaries lead to iteration and rework (as seen in Task 05's retry).

#### 4. Integration Tests, Not Mocks
FastAPI's TestClient should run against real middleware and real handlers, not mocks. Mock/prod divergence is a hidden risk that external reviewers would catch late.

### Phase 0 Deliverables

The Phase 0 backend server should include:

1. **FastAPI Server Scaffold**
   - Minimal core with well-defined extension points
   - Middleware hooks for instrumentation, auth, rate-limiting
   - Clean separation of concerns

2. **Token Ledger Integration**
   - Consumer API for querying token usage by model, task, connector
   - Streaming interface for real-time metrics
   - Cost attribution and budget tracking

3. **Connector Abstraction Layer**
   - Unified interface for Claude, OpenCode, GPT, etc.
   - Session management and caching
   - Error normalization and retry handling

4. **WebSocket Subscription Support**
   - Live mission state updates
   - Event streaming (task started, approved, failed, etc.)
   - Dashboard real-time sync

5. **State Persistence**
   - JSON-based mission state (filesystem or DB backend)
   - Atomic writes with versioning
   - Resumption after crash

## Appendix A: Code Principles Framework

This analysis applies the following Code Principles to evaluate software quality and guide structural decisions. When principles conflict, follow the priority order below.

### When to Skip These Principles

- One-off scripts, spikes, or throwaway prototypes (unless they handle user data or secrets — Safety First always applies)
- Pure test code (test readability trumps DRY — some duplication is acceptable for clarity)
- Trivial changes (renaming, formatting, dependency bumps)

### User Intent Overrides

These priorities govern the AI's default reasoning. If the user explicitly states a requirement or constraint that conflicts with the priority order, the user's stated intent takes precedence.

### Priority Order

When principles conflict, the higher-priority principle wins:

1. **Safety First** — Security, data integrity, correctness
2. **YAGNI** — Don't build what you don't need yet
3. **Occam's Razor** — Prefer the simplest solution that works
4. **SOLID** — Structure complex systems for maintainability
5. **DRY** — Eliminate duplicated knowledge (not at the cost of clarity)
6. **Miller's Law** — Respect the 7±2 cognitive limit in APIs, parameters, and module boundaries

### Quick Decision Questions

Before writing or approving code, ask:

- "Is there a simpler way?" → Occam's Razor
- "Understandable in under 1 minute?" → Miller's Law
- "Am I duplicating knowledge?" → DRY
- "Is this needed right now?" → YAGNI
- "Is this safe by default?" → Safety First

### Detailed Principles

#### 1. Safety First

Never compromise security or data integrity for elegance. Safety encompasses:
- Parameterize all queries (SQL, NoSQL, ORM) to prevent injection attacks
- Validate and sanitize inputs at system boundaries, not deep in the stack
- Never log secrets, tokens, or PII
- Use allowlists over denylists for input filtering
- Default to deny/closed/private (fail-safe by default)
- Audit dependencies for known vulnerabilities

**Example**: A handler that validates user access:
```python
# BAD — trusts caller input for authorization
def get_user(user_id: str):
    return db.users.find(user_id)

# GOOD — validates access at the boundary
def get_user(user_id: str, requester_id: str):
    user = db.users.find(user_id)
    if user.owner_id != requester_id:
        raise ForbiddenError()
    return user
```

In the HTTP server context: error handlers normalize exceptions (never expose internal stack traces), environment variables protect secrets, parameterized ORM queries prevent SQL injection.

---

#### 2. YAGNI — You Aren't Gonna Need It

Build for today's requirements. Remove speculative code. If a requirement is confirmed and scheduled, it's not speculative — plan for it.

**Example**:
```python
# BAD — supports 3 export formats when only CSV is needed
class ExportStrategy: pass
class CsvExport(ExportStrategy): pass
class JsonExport(ExportStrategy): pass
class XmlExport(ExportStrategy): pass

# GOOD — just do what's needed
def export_csv(data: List[Row]) -> bytes:
    return "\n".join(",".join(str(x) for x in row) for row in data).encode()
```

When you need JSON export later, add it then — not now.

**Exception — known abstraction boundaries**: External service integrations (payment providers, email, storage, auth) are well-known volatility points. A thin interface at these boundaries is justified even with a single implementation.

In the HTTP server: No auth layer, no caching, no observability agent added speculatively. Only the 5 required features (scaffold, metrics, error handling, tests, Docker).

---

#### 3. Occam's Razor — Simplest Solution

Given two solutions that solve the problem, pick the simpler one. Simplicity reduces cognitive load, bugs, and maintenance burden.

**Example**:
```python
# BAD — over-engineered event system for a single notification
class EventBus:
    def publish(self, event: str, data: dict): pass
    def subscribe(self, event: str, handler): pass

event_bus.publish("user.created", user)

# GOOD — direct call when there's one consumer
async def on_user_created(user: User):
    await send_welcome_email(user.email)
```

Add the event bus when you actually have multiple consumers.

In the HTTP server: Error handler uses direct exception mapping (not a middleware chain), metrics via simple decorator, tests use pytest fixtures over complex factories.

---

#### 4. SOLID — For Complex Systems

Apply when the system has real complexity that justifies the structure. This section focuses on **Single Responsibility** and **Dependency Inversion** — the most frequently relevant.

**Single Responsibility** — One reason to change per module.

```python
# BAD — handler does validation, business logic, and persistence
async def handle_order(request: Request):
    if not request.json.get("items"):
        raise ValueError("empty")
    total = sum(item["price"] for item in request.json["items"])
    await db.orders.insert({"items": request.json["items"], "total": total})
    await email_service.send(request.json["email"], total)

# GOOD — each concern is separated
async def handle_order(request: Request):
    input_data = validate_order_input(request.json)
    order = create_order(input_data)
    await save_order(order)
    await notify_customer(order)
```

**Dependency Inversion** — Depend on abstractions at system boundaries.

```python
# GOOD — swappable storage behind an interface
class OrderStore(Protocol):
    async def save(self, order: Order) -> None: ...

def create_order_service(store: OrderStore):
    async def place(input_data: OrderInput) -> Order:
        order = create_order(input_data)
        await store.save(order)
        return order
    return place
```

But don't create interfaces for internal code with a single implementation — that's YAGNI.

In the HTTP server: Metrics isolated from error handling, connectors injectable via dependency injection, test fixtures demonstrate loose coupling.

---

#### 5. DRY — Don't Repeat Knowledge

Eliminate duplication of **knowledge**, not just characters. Three similar lines are fine if they represent different concepts.

**Example**:
```python
# BAD — duplicated validation rule
def validate_on_create(email: str):
    if "@" not in email:
        raise ValueError("invalid")

def validate_on_update(email: str):
    if "@" not in email:
        raise ValueError("invalid")

# GOOD — single source of truth
def validate_email(email: str):
    if "@" not in email:
        raise ValueError("invalid email")
```

```python
# DON'T over-DRY — these look similar but represent different concepts
user_query = "SELECT id, name FROM users WHERE active = true"
admin_query = "SELECT id, name FROM admins WHERE active = true"
# Merging into build_active_query(table) adds coupling for no real gain
```

In the HTTP server: Error codes defined once in config.py, test assertions consolidated in fixtures. Some test setup is repeated (acceptable for clarity).

---

#### 6. Miller's Law — 7±2 Cognitive Limit

Functions, configs, and APIs should not exceed what a person can hold in working memory at once.

**Example**:
```python
# BAD — 11 parameters, no one remembers the order
def create_user(name, email, age, role, dept, manager, start_date, salary, currency, location, timezone):
    pass

# GOOD — grouped into a coherent object
class CreateUserInput:
    name: str
    email: str
    role: Role
    department: Department
    start_date: date

def create_user(input: CreateUserInput):
    pass
```

**Rule of thumb**: Flag when a function exceeds 5 parameters, a module exposes more than 7 public functions, or a conditional has more than 5 branches.

In the HTTP server: Main handler functions average 3 parameters, middleware configs use 4 fields, error enum has ~10 members (grouped, acceptable).

### Conflict Resolution Examples

| Situation | Winning Principle | Reasoning |
|-----------|-------------------|-----------|
| "Add an abstraction layer for future DB swap" | YAGNI over SOLID | No current need for the swap |
| "Inline the SQL to keep it simple" but it handles user input | Safety First over Occam's | Must parameterize queries regardless of simplicity |
| "Extract a helper" but it's used once | YAGNI over DRY | One callsite doesn't justify extraction |
| "Split into 12 micro-modules" | Miller's Law over SOLID | Too many modules defeats comprehension |
| "Copy-paste is simpler" but it's a business rule | DRY over Occam's | Duplicated business rules diverge and cause bugs |

---

### Applying During Planning

When reviewing or creating a plan, validate each component against:

1. Does any part introduce a security risk? → Fix it regardless of other tradeoffs
2. Is any part speculative or not required by current scope? → Remove it
3. Is there a simpler design that satisfies the requirements? → Use it
4. For remaining complexity, is the structure maintainable? → Apply SOLID
5. Is any knowledge duplicated? → Consolidate if it won't hurt clarity
6. Can someone new understand each piece quickly? → Simplify interfaces

### Language-Specific Notes

These principles are language-agnostic, but their expression varies:

- **Rust**: The compiler enforces memory safety; focus Safety First on logic errors and `unsafe` blocks.
- **Go**: Prefer concrete types until a second implementation exists (YAGNI). Interfaces are implicit and cheap — use them at consumption site.
- **Python**: Use `typing.Protocol` over ABC inheritance for Dependency Inversion. Avoid metaprogramming as a DRY mechanism when it hurts readability.

Adapt examples to the target language's idioms.
